## Spacing Scale

Inherited from the existing design system. All values remain multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Dot indicators, inline icon gaps |
| sm | 8px | Compact spacing inside pills and badges |
| md | 16px | Default row padding, bubble padding, section spacing |
| lg | 24px | Header padding, panel padding |
| xl | 32px | Desktop pane gap, empty-state padding |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**Exceptions for Phase 6:**
- Mobile back button, thread switcher, and message send controls: minimum 44px touch target
- Conversation row vertical rhythm: 12px internal spacing for dense scanning
- Message composer: minimum 44px input height, 56px total sticky action bar height on mobile

## Color

Inherit the current white/slate chat treatment from `frontend/src/index.css` and the existing chat surfaces.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | oklch(1 0 0) / `#ffffff` | Page background, primary chat surfaces |
| Secondary (30%) | oklch(0.97 0 0) / `#f4f4f5` | Conversation list, cards, secondary panels |
| Accent (10%) | oklch(0.205 0 0) / `#18181b` | Active thread, own message bubble, high-priority controls |
| Destructive | oklch(0.577 0.245 27.325) / `#dc2626` | Destructive actions only |

**Accent reserved for:**
- Active conversation row container
- Current user's message bubble
- Primary mobile navigation controls inside the messages shell
- Keyboard focus rings
- High-priority unread emphasis when the row is active

**Phase 6 semantic additions:**

| Element | Value | Usage |
|---------|-------|-------|
| Live presence | emerald-500 | Seller online dot and live transport success state |
| Status syncing | amber-500 | Presence stale, reconnecting, degraded realtime state |
| Status unavailable | slate-400 | Presence temporarily unknown after stale timeout |
| Received message bubble | white with slate border | Other-party messages |

**Critical rule:** transport state and seller presence are separate signals. A websocket reconnect must not recolor the seller as offline by default.

## Visual Focal Points

### Primary Focal Points

| Screen | Primary Focal Point | Visual Treatment |
|--------|--------------------|------------------|
| Messages list | Active conversation row | Accent border/background, strongest contrast in the list |
| Active thread | Seller name + presence pill | Header cluster stays pinned at top of thread |
| Mobile thread view | Sticky header controls | Back/thread-switch control remains visible while scrolling |
| Reconnect state | Transport banner/helper text | Amber status treatment, never larger than one compact line above composer |

### Hierarchy Rules

1. The selected conversation is the visual anchor in list mode.
2. In thread mode, the seller identity and status cluster is the anchor.
3. Reliability feedback is secondary and compact; it must not overpower actual message content.

## Interaction Patterns

### Responsive Layout Contract

**Route:** `/messages`

**Desktop (`>= 1024px`)**
- Two-pane layout is required.
- Conversation list width: 320px fixed.
- Active thread pane: fills remaining width.
- Both panes remain visible at all times.

**Tablet (`768px-1023px`)**
- Two-pane layout remains allowed.
- Conversation list width reduces to 280px.
- Thread header and composer padding stay at `md` or `lg`; never tighter than 16px.

**Mobile (`<= 767px`)**
- Single-pane layout is required.
- Default view is the conversation list.
- Selecting a conversation replaces the list with the thread view in the same route state.
- Thread header must include a visible back control labeled or aria-labeled "Back to conversations".
- Composer stays sticky to the bottom edge and respects safe-area padding.

### Presence Reliability Contract

1. Seller presence and transport connection state must be rendered separately.
2. On websocket disconnect, preserve the last known seller presence for 30 seconds.
3. During that 30-second window, show the transport pill as `Connecting` or `Reconnecting`, but do not switch seller status to `Offline`.
4. After 30 seconds without a fresh presence update, replace seller presence copy with `Status updating`.
5. Only render `Last seen {relative time}` when the backend explicitly provides an offline presence payload.
6. `Seller online` transitions may trigger a toast once per seller session, not once per conversation row.

### Multi-Conversation Seller Sync Contract

1. All conversation rows for the same `otherUserId` must update in the same render cycle.
2. The active thread header and every matching list row must show the same presence copy and dot color.
3. When a seller appears in multiple listing conversations, unread counts remain per conversation, but presence remains per seller.
4. Repeated-seller rows must never disagree between `online`, `last seen`, and `status updating`.

### Message Layout Contract

1. Own message bubbles align right; received bubbles align left.
2. Desktop bubble max width: 70% of thread width.
3. Mobile bubble max width: 85% of thread width.
4. Long words, URLs, and image attachments must wrap without horizontal scrolling.
5. Timestamps remain inside the bubble footer and never wrap onto a third line by themselves.
6. Typing indicator stays between message list and composer and must not overlap the sticky composer.

### Realtime Fallback Contract

1. If realtime transport is unavailable, message sending falls back to REST without changing the send button label.
2. Fallback mode displays helper copy above the composer, not a modal or full-width blocking banner.
3. Periodic refresh keeps conversation previews, unread counts, and presence text synchronized after reconnect.
4. Redis-backed fan-out must feel invisible in the UI: no duplicate list updates, no repeated presence flicker, no row reordering unless `lastMessageAt` changes.

### Connection States

| State | Transport UI | Seller Presence UI |
|-------|--------------|--------------------|
| Connected | Compact live pill with emerald icon | Show `Seller online` or `Last seen ...` from latest presence |
| Connecting | Compact neutral/amber pill | Preserve last known seller state |
| Reconnecting | Amber helper copy above composer | Preserve last known seller state for 30s |
| Stale presence | Transport state unchanged | Replace seller state with `Status updating` |
| Offline fallback | Amber helper copy above composer | Keep `Status updating` until backend sends offline or online presence |

## Accessibility

| Requirement | Implementation |
|-------------|----------------|
| Keyboard navigation | Full tab flow through list rows, back control, composer, send action |
| Screen reader | Announce presence changes as polite live-region updates, not assertive interruptions |
| Mobile navigation | Back control must have visible label or descriptive aria-label |
| Color contrast | WCAG AA minimum for pills, badges, and helper text |
| Status indicators | Presence and transport must use icon plus text, never color alone |
| Reduced motion | Do not animate pane transitions more than 200ms; disable non-essential motion when reduced motion is requested |
